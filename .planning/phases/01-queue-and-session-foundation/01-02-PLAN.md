    ```sql
    -- Migration 012: sam_sessions table for cross-restart session persistence
    -- Replaces volatile in-memory sessions and conversations Maps in both gateways
    -- Key scheme: "platform:chatId" (e.g. "telegram:1385723011", "whatsapp:+13105551234")

    CREATE TABLE IF NOT EXISTS sam_sessions (
      session_key    text PRIMARY KEY,           -- "telegram:1385723011"
      platform       text NOT NULL,              -- "telegram" | "whatsapp"
      chat_id        text NOT NULL,              -- platform-native ID as string
      last_estimate  jsonb,                      -- full buildEstimate() results object
      history        jsonb DEFAULT '[]'::jsonb,  -- Claude messages array (capped at 20 by gateway)
      stage          text DEFAULT 'idle',        -- "idle" | "collecting_info" | "queued" | "running" | "done"
      collected_data jsonb DEFAULT '{}'::jsonb,  -- { name, phone, vehicle, complaint }
      created_at     timestamptz DEFAULT now(),
      updated_at     timestamptz DEFAULT now()
    );

    -- Index for 24h TTL cleanup sweep (DELETE WHERE updated_at < now() - interval '24 hours')
    CREATE INDEX IF NOT EXISTS sam_sessions_updated_at_idx ON sam_sessions (updated_at);

    -- Comment documenting the design decision
    COMMENT ON TABLE sam_sessions IS
      'Persistent conversation sessions for SAM gateways. session_key is the primary lookup. '
      'last_estimate and history are JSONB to avoid schema churn as estimate results evolve. '
      'Stage field is text (not enum) to avoid migrations when new stages are added in Phase 2.';
    ```

    Create `skills/shared/session-store.js` following the exact pattern from RESEARCH.md Pattern 2 (Supabase Session Store with In-Memory Fallback).

    Key implementation details:

    1. **Module-level config constants:**
       ```javascript
       const TABLE = "sam_sessions";
       const TTL_MS = 24 * 60 * 60 * 1000; // 24 hours
       ```

    2. **Supabase client init:** Create only if both `SUPABASE_URL` and `SUPABASE_ANON_KEY` are set. If either is missing, `supabase = null` — all Supabase operations are silently skipped. Do NOT throw.

    3. **memCache write-through:** `setSession` writes to `memCache` synchronously (before await) so subsequent synchronous getSession callers from the same process get fresh data instantly.

    4. **getSession cold start:** On cache miss, try Supabase. Map DB columns (`last_estimate`, `history`, `stage`, `collected_data`) back to the session object shape. Update memCache so the next call is a cache hit.

    5. **setSession upsert:** Use `{ onConflict: "session_key" }`. Set `updated_at: new Date().toISOString()` explicitly. Write both `last_estimate` and `history` in every upsert call — do not write them separately.

    6. **cleanupExpiredSessions:** Build cutoff as `new Date(Date.now() - TTL_MS).toISOString()`. Use `.lt("updated_at", cutoff)`. Chain `.select("session_key")` to get count of deleted rows. Return `{ deleted: count }`.

    7. **Exports:** `module.exports = { getSession, setSession, deleteSession, cleanupExpiredSessions, makeKey };`

    Note from research — avoid this pattern:
    - Do NOT store history separately from lastEstimate. Both go in the same upsert call.
    - Do NOT throw on Supabase errors — log with `${LOG}` prefix and continue.
